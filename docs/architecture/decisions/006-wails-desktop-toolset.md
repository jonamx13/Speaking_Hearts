# ADR 006: Selection of Wails Framework for Post-Event Desktop Utility

## Status
Accepted

## Context
Following the conclusion of a ceremonial event, the **Speaking_Hearts** system generates a significant volume of fragmented data, including audio segments and structured JSON metadata. To provide value to professional videographers and event organizers, a dedicated utility is required to rebuild these fragments into coherent, time-synchronized assets (e.g., SubRip Subtitle tracks). This utility must be a standalone desktop application capable of operating in the same offline, resource-constrained environments as the primary engine.

## Decision
We have selected the **Wails** framework (Go + Web Technologies) for the development of the Phase 7 Post-Event Toolset.

## Rationale

### 1. Backend Logic Portability and Code Reuse
A primary architectural driver for this selection is the ability to directly reuse the core Go logic implemented in the processing and storage layers. Specifically, the complex JSON unmarshaling, file system traversal, and data reconciliation logic used by the **AudioRebuilder** (implemented in `cmd/posttool/rebuild`) can be integrated into the desktop binary without modification. This guarantees that the post-event utility maintains perfect parity with the data formats generated by the live engine.

### 2. Native Performance vs. Electron
Unlike the Electron framework, which bundles a complete Chromium browser and Node.js runtime—imposing significant memory and disk space overhead—Wails utilizes the host operating system's native webview. This results in a significantly smaller binary footprint (often <10MB) and lower runtime resource consumption. For videographers who may already be running resource-intensive editing software, this lightweight footprint is a critical performance requirement.

### 3. Native File System Integration
Wails provides a streamlined bridge between the web-based frontend and Go’s powerful system-level capabilities. This facilitates a superior user experience for the required "folder-picker" workflow, allowing for high-performance scanning and processing of the `storage/` directory directly within the native OS environment.

## User Experience (UX) Requirements
The utility will adhere to a functional, high-utility design pattern:
*   **Input Interface:** A native OS directory selection prompt targeting the primary or backup `storage/` path.
*   **Processing Engine:** Automated reconciliation of the `metadata.json` timeline across all ceremony fragments.
*   **Output Result:** The generation of multi-language, time-synchronized `.srt` subtitle files, ready for immediate import into professional video editing software (e.g., Adobe Premiere, DaVinci Resolve).

## Consequences
*   The build environment must include the Wails CLI and the required native development headers for the target OS (Windows/Linux/macOS).
*   The team must maintain a clean separation between the "Rebuilder" core logic and the Wails UI bindings to ensure long-term modularity.
